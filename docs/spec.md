# AI 확장 대비 게시판 구현 스펙

## 1. 프로젝트 목적

이 프로젝트는 일반 게시판을 먼저 완성한 뒤, 공지사항, FAQ, 과거 질문/답변 데이터를 기반으로 RAG, GraphRAG, MCP 기반 지식 베이스형 Q&A 봇으로 확장할 수 있게 설계한다.

1차 목표는 게시판의 핵심 흐름을 완성하는 것이다. 회원가입, 로그인, 게시글 CRUD, 댓글, 좋아요, 태그, 검색, 페이지 번호, 이미지 업로드, 관리자 공지/FAQ 기능을 포함한다.

## 2. 기술 스택

- 프로젝트 위치: `게시판/front`, `게시판/backend`
- 백엔드: Java 21, Spring Boot, Spring Security, Spring Data JPA, PostgreSQL
- 프론트엔드: Vite, React, TypeScript, react-router-dom
- 테스트: JUnit, Spring Boot Test, H2 PostgreSQL mode, Vitest, React Testing Library
- DB 마이그레이션: Flyway
- API 문서: Swagger
- 이미지 저장: `FileStorage` 인터페이스 + 1차 `LocalFileStorage`
- 스타일: UI 라이브러리 없이 CSS 직접 구현

## 3. 확정된 트레이드오프

- OpenAI API 기반 태그 추천은 비용이 발생하므로 1차 구현에서 제외한다.
- 태그는 수동 입력을 기본으로 하고, 로컬 키워드 후보 추천을 보조 기능으로 둔다.
- 게시글 목록은 인피니티 스크롤이 아니라 페이지 번호 방식을 사용한다.
- API의 페이지 번호는 사용자 기준 `page=1`부터 시작하고, 내부에서만 `page - 1`로 변환한다.
- 게시글과 댓글은 하드 삭제가 아니라 소프트 삭제한다.
- 첫 번째 가입자는 `ADMIN`, 이후 가입자는 `USER`로 만든다.
- Refresh Token은 사용자당 활성 토큰 1개만 허용한다.
- 테스트 DB는 H2 PostgreSQL mode를 사용하고, 실제 실행은 Docker PostgreSQL을 사용한다.
- 이미지 파일은 1차에서 로컬 저장소에 저장하고, S3 연결은 추후 `S3FileStorage`로 확장한다.

## 4. 기능 요구사항

### 회원가입 / 로그인

- 이메일, 비밀번호, 닉네임, 프로필 이미지를 입력받는다.
- 프로필 이미지는 회원가입 시 필수다.
- 이메일과 닉네임은 중복될 수 없다.
- 비밀번호는 8자 이상 20자 이하이며, 대문자, 소문자, 숫자, 특수문자를 각각 최소 1개 포함한다.
- 로그인 실패 시 `아이디 또는 비밀번호를 확인해주세요` 메시지를 사용한다.

### 인증 / 권한

- Access Token + Refresh Token 구조를 사용한다.
- Refresh Token은 DB에 해시로 저장한다.
- Refresh Token은 HttpOnly Cookie로 전달한다.
- Access Token은 프론트에서 API 요청에 사용한다.
- 관리자만 공지와 FAQ를 작성, 수정, 삭제할 수 있다.
- 일반 사용자는 일반글과 질문글을 작성할 수 있다.
- 게시글 수정/삭제는 작성자 또는 관리자만 가능하다.

### 게시글

- 게시글 타입은 `GENERAL`, `NOTICE`, `FAQ`, `QUESTION`을 사용한다.
- 제목은 최대 26자까지만 작성 가능하다.
- 본문 길이 제한은 없다.
- 게시글 이미지는 1개만 업로드할 수 있다.
- 게시글 목록에는 번호, 분류, 제목, 태그, 작성자, 댓글 수, 좋아요 수, 조회 수, 작성일을 표시한다.
- 날짜 형식은 `yyyy-MM-dd HH:mm:ss`를 사용한다.
- 숫자 표기는 1,000 이상 `1k`, 10,000 이상 `10k`, 100,000 이상 `100k`를 사용한다.
- 삭제된 게시글은 목록, 검색, RAG 대상에서 제외한다.

### 댓글

- 댓글 내용은 TEXT로 저장한다.
- 댓글은 소프트 삭제한다.
- 질문글에는 채택 댓글을 지정할 수 있도록 `acceptedCommentId` 확장을 고려한다.

### 태그

- 태그는 사용자가 직접 입력한다.
- 중복 태그는 하나로 관리한다.
- 로컬 태그 후보 추천은 제목, 본문, 기존 태그를 기준으로 제공한다.
- 태그 추천은 외부 API를 호출하지 않는다.

### 좋아요

- 로그인한 사용자는 게시글 좋아요를 토글할 수 있다.
- 같은 사용자는 같은 게시글에 좋아요를 1번만 누를 수 있다.
- 활성 상태에서 다시 클릭하면 좋아요가 취소된다.

### 검색 / 페이징

- 검색 대상은 제목, 본문, 작성자 닉네임, 태그다.
- 1차 검색은 DB `LIKE` 검색으로 구현한다.
- 기본 페이지 값은 `page=1`, `size=10`이다.
- 목록은 페이지 번호 방식으로 이동한다.

### 이미지

- 허용 확장자는 `jpg`, `jpeg`, `png`, `webp`다.
- 최대 파일 크기는 5MB다.
- 실제 파일 삭제 자동화는 1차 구현에서 제외한다.

## 5. 프론트 화면 요구사항

- 라우트는 `/login`, `/signup`, `/posts`, `/posts/:postId`, `/posts/new`, `/posts/:postId/edit`, `/profile/edit`, `/password/edit`을 사용한다.
- UI는 디시/에펨형 게시판처럼 표 기반 목록을 중심으로 한다.
- 상단에는 검색창, 분류 필터, 태그 필터, 로그인/프로필 버튼, 글쓰기 버튼을 배치한다.
- 모바일에서는 표를 억지로 줄이지 않고 제목 중심 리스트로 전환한다.
- 로그인, 회원가입, 프로필 수정, 비밀번호 수정 유효성 메시지는 사용자가 정의한 문구를 그대로 사용한다.
- 게시글 작성/수정 화면에서 제목은 26자 초과 입력이 되지 않게 한다.
- 이미지 업로드 시 미리보기를 제공한다.

## 6. TDD 및 테스트 범위

백엔드는 TDD를 강하게 적용한다. 실패 테스트를 먼저 작성하고, 최소 구현으로 통과시킨 뒤 리팩터링한다.

백엔드 테스트 범위:

- 회원가입 유효성
- 중복 이메일/닉네임
- 첫 가입자 ADMIN, 이후 USER
- 로그인 성공/실패
- Refresh Token 재발급/만료
- 작성자와 관리자 권한
- 관리자만 공지/FAQ 작성 가능
- 게시글 CRUD
- 게시글 검색/분류/태그 필터/페이지 번호
- 댓글 등록/삭제/채택
- 좋아요 토글 중복 방지
- 소프트 삭제 데이터 제외
- 태그 수동 입력/중복 방지
- 로컬 태그 후보 추천
- 이미지 확장자/크기 검증

프론트 테스트는 핵심 유효성과 핵심 컴포넌트 위주로 작성한다.

- 로그인/회원가입 유효성 메시지
- 제목 26자 제한
- 게시글 목록 페이지 번호 이동
- 검색/필터 상태 유지
- 좋아요 토글 UI
- 이미지 업로드 미리보기
- 프로필/비밀번호 수정 유효성
- 모바일 목록 핵심 레이아웃

## 7. RAG / GraphRAG / MCP 확장 대비

1차 DB부터 다음 필드를 고려한다.

- `postType`
- `canonicalUrl`
- `isPublic`
- `publishedAt`
- `deletedAt`
- `acceptedCommentId`

RAG 2차 확장:

- PostgreSQL + `pgvector` 기반 일반 RAG를 먼저 구현한다.
- 공지, FAQ, 질문/답변 게시글을 chunk로 분리한다.
- 답변에는 원문 제목, 링크, 근거 내용을 포함한다.

GraphRAG 3차 확장:

- 게시글, 태그, 이벤트, 날짜, 작성자, 질문, 답변 관계를 그래프로 연결한다.
- 예: `작년 이벤트 당첨자 발표 언제였어?` 같은 질문에 관련 공지 링크와 내용을 반환한다.

MCP 4차 확장:

- 기존 API를 기반으로 MCP tool을 감싼다.
- 후보 도구는 `searchPosts`, `getPostDetail`, `searchNotices`, `searchFaq`, `askKnowledgeBase`다.
