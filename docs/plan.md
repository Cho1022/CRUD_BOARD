# 구현 마일스톤 계획

## 마일스톤 1. 프로젝트 골격 생성

### 목표

기존 루트 프로젝트를 건드리지 않고 `게시판/front`, `게시판/backend`에 새 프로젝트를 만든다.

### 구현 항목

- Spring Initializr 또는 IntelliJ 생성 기능으로 Java 21 Spring Boot 프로젝트 생성
- Gradle Wrapper 포함
- Vite React TypeScript 프로젝트 생성
- 새 Docker Compose 설정은 `게시판` 하위에만 작성
- PostgreSQL 실행 설정과 기본 환경 변수 구성

### TDD 테스트 항목

- 백엔드 애플리케이션 컨텍스트 로딩 테스트
- 프론트 기본 렌더링 테스트

### 완료 기준

- 백엔드 테스트 명령이 실행된다.
- 프론트 테스트 명령이 실행된다.
- 기존 루트 `backend`, `frontend`, `docker-compose.yml`은 변경되지 않는다.

### 주의할 점

- 의존성 다운로드나 네트워크 문제가 생기면 중단하고 사용자와 상의한다.
- 전역 Gradle, Node, Java 설정은 변경하지 않는다.

## 마일스톤 2. 백엔드 공통 기반

### 목표

모든 도메인이 공유할 응답, 예외, 감사 필드, 보안 기본 틀을 만든다.

### 구현 항목

- `ApiResponse`, `PageResponse`, 공통 예외, 에러 코드 작성
- 생성일, 수정일, 삭제일 공통 필드 구성
- Flyway 기본 마이그레이션 구성
- Swagger 설정
- H2 PostgreSQL mode 테스트 설정

### TDD 테스트 항목

- 공통 응답 형태 테스트
- 예외 응답 테스트
- 페이지 번호 `page=1` 입력 시 내부 `0` 변환 테스트

### 완료 기준

- 공통 응답 구조가 모든 API에서 재사용 가능하다.
- 테스트 DB와 실제 PostgreSQL 설정이 분리된다.

### 주의할 점

- Controller에 비즈니스 로직을 넣지 않는다.
- 응답 구조를 기능마다 다르게 만들지 않는다.

## 마일스톤 3. 인증/회원 TDD

### 목표

회원가입, 로그인, Refresh Token, 권한의 핵심 흐름을 완성한다.

### 구현 항목

- `Member`, `RefreshToken`, 권한 enum 작성
- 회원가입, 로그인, 로그아웃, 토큰 재발급, 내 정보 조회 API 구현
- 첫 가입자 `ADMIN`, 이후 가입자 `USER` 정책 구현
- Refresh Token 해시 저장
- HttpOnly Cookie 기반 Refresh Token 전달

### TDD 테스트 항목

- 회원가입 유효성
- 중복 이메일/닉네임
- 첫 가입자 ADMIN, 이후 USER
- 로그인 성공/실패
- Refresh Token 재발급/만료

### 완료 기준

- 인증 관련 핵심 테스트가 통과한다.
- 로그인 실패 메시지가 명세와 일치한다.

### 주의할 점

- 사용자당 활성 Refresh Token은 1개만 허용한다.
- 비밀번호는 반드시 해시로 저장한다.

## 마일스톤 4. 게시글 CRUD TDD

### 목표

게시판의 중심 기능인 게시글 작성, 조회, 수정, 삭제, 목록을 완성한다.

### 구현 항목

- `Post`, 게시글 타입, 소프트 삭제 구현
- 게시글 작성, 상세 조회, 수정, 삭제 API
- 목록 API에 검색, 분류, 페이지 번호 적용
- 제목 26자 제한
- 조회수 증가

### TDD 테스트 항목

- 게시글 작성 성공/실패
- 제목 26자 초과 실패
- 작성자만 수정/삭제 가능
- 관리자는 수정/삭제 가능
- 삭제된 게시글 목록/검색 제외
- 페이지 번호와 검색 조건 테스트

### 완료 기준

- 게시글 CRUD 테스트가 통과한다.
- 목록 API가 `page=1`, `size=10` 기준으로 동작한다.

### 주의할 점

- 삭제는 하드 삭제가 아니라 소프트 삭제다.
- RAG 확장을 위해 `postType`, `canonicalUrl`, `isPublic`, `publishedAt`을 고려한다.

## 마일스톤 5. 댓글/좋아요/태그

### 목표

게시글 상호작용 기능을 완성한다.

### 구현 항목

- 댓글 등록, 조회, 삭제 구현
- 질문글 채택 댓글 확장 지점 구성
- 좋아요 토글 구현
- 태그 수동 입력과 중복 방지 구현
- 로컬 키워드 기반 태그 후보 추천 구현

### TDD 테스트 항목

- 댓글 등록/삭제
- 삭제된 댓글 제외
- 좋아요 중복 방지와 취소
- 태그 중복 방지
- 제목/본문/기존 태그 기반 후보 추천

### 완료 기준

- 댓글, 좋아요, 태그 핵심 테스트가 통과한다.
- 태그 추천은 외부 API 없이 동작한다.

### 주의할 점

- 태그 추천은 보조 기능이며 게시글 작성을 막지 않는다.
- OpenAI API나 외부 유료 API를 호출하지 않는다.

## 마일스톤 6. 이미지 업로드

### 목표

프로필 이미지와 게시글 이미지 업로드를 로컬 저장소로 구현하고 S3 확장 지점을 만든다.

### 구현 항목

- `FileStorage` 인터페이스 작성
- `LocalFileStorage` 구현
- 프로필 이미지 필수 업로드
- 게시글 이미지 1개 업로드
- 파일 확장자와 크기 검증

### TDD 테스트 항목

- 허용 확장자 업로드 성공
- 미허용 확장자 실패
- 5MB 초과 실패
- 프로필 이미지 누락 실패
- 게시글 이미지 2개 이상 방지

### 완료 기준

- 이미지 검증 테스트가 통과한다.
- 저장소 구현 교체가 가능한 구조다.

### 주의할 점

- S3 연결은 1차 구현에 포함하지 않는다.
- 게시글 삭제 시 실제 이미지 파일 자동 삭제는 1차에서 제외한다.

## 마일스톤 7. 관리자 공지/FAQ

### 목표

RAG 지식베이스의 기반이 될 신뢰도 높은 게시글 타입을 관리한다.

### 구현 항목

- `NOTICE`, `FAQ` 게시글 작성 권한 제한
- 관리자 전용 수정/삭제 정책 적용
- 공지/FAQ 필터링 목록 제공

### TDD 테스트 항목

- 일반 사용자의 공지/FAQ 작성 실패
- 관리자의 공지/FAQ 작성 성공
- 공지/FAQ 검색과 필터

### 완료 기준

- 관리자 권한 정책이 테스트로 검증된다.
- 공지/FAQ가 일반글과 구분된다.

### 주의할 점

- 공지/FAQ는 추후 RAG 근거 데이터로 사용되므로 소프트 삭제와 공개 여부를 지킨다.

## 마일스톤 8. 프론트 화면 연결

### 목표

주요 화면을 API와 연결하고 디시/에펨형 게시판 UX를 완성한다.

### 구현 항목

- 로그인, 회원가입, 게시글 목록, 상세, 작성, 수정 화면 구현
- 프로필 수정, 비밀번호 수정 화면 구현
- 검색창, 분류 필터, 태그 필터, 페이지 번호 UI 구현
- 이미지 업로드 미리보기 구현
- 모바일 제목 중심 리스트 구현

### TDD 테스트 항목

- 로그인/회원가입 유효성 메시지
- 제목 26자 제한
- 페이지 번호 이동
- 검색/필터 상태 유지
- 좋아요 토글 UI
- 이미지 업로드 미리보기

### 완료 기준

- 주요 화면에서 백엔드 API 흐름이 연결된다.
- 핵심 프론트 테스트가 통과한다.

### 주의할 점

- 프론트는 모든 동작을 과도하게 테스트하지 않고 유효성/핵심 컴포넌트 중심으로 테스트한다.
- UI 라이브러리 없이 CSS로 구현한다.

## 마일스톤 9. 테스트/검증/정리

### 목표

1차 완료 기준을 검증하고 다음 RAG 확장으로 넘어갈 수 있는 상태로 정리한다.

### 구현 항목

- 백엔드 전체 테스트 실행
- 프론트 핵심 테스트 실행
- 실제 PostgreSQL 실행 검증
- Swagger API 확인
- README 실행 방법 정리
- RAG/MCP 확장 메모 정리

### TDD 테스트 항목

- 전체 회귀 테스트
- 인증부터 게시글 작성, 댓글, 좋아요, 검색까지 핵심 흐름 통합 테스트

### 완료 기준

- 1차 완료 기준이 모두 충족된다.
- 기존 루트 프로젝트가 오염되지 않았다.
- 다음 단계인 RAG/GraphRAG/MCP 확장 지점이 문서화되어 있다.

### 주의할 점

- Docker 볼륨 삭제, 기존 컨테이너 삭제, Git push 같은 위험 작업은 하지 않는다.
